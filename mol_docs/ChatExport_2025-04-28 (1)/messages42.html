Обход дерева не дорогой, тут скорость зависит от того какая часть кеша используется неэффективно. Процессоры работают таким образом что на любое обращение в память происходит загрузка сразу всей кеш-линии (обычно это 64 байта в современных процессорах). То есть если прочитал хотя бы один байт - получай в бонус бесплатное чтение рядом лежащих 64 байта (они будут считываться уже из кеша на порядок быстрее чем из памяти). А поскольку работа с массивами обычно такая что происходит чтение данных последовательно то получается что с массивами кеш используется эффективнее когда же со связанными списками мы будем как бы прыгать по кеш-линиям - загрузили один объект который занимает лишь небольшую часть кеш линии а для следующего объекта придется грузить уже новую кеш-линию и т.д. Но с другой стороны если дерево или связанные списки организовать таким образом чтобы элемент связанного списка (или то что мы хотим читать) имел размер 64 байта то тогда уже получается что даже со связанными списками будем использовать кеш эффективно и скорость будет сравнима с проходом по массиву. С другой стороны не надо считать что использование массива гарантирует скорость - все зависит от того что мы читаем - если например данные находятся в массиве но в процессе прохода мы читаем только одно поле из жирного объекта (например нужно найти какой-то один объект) то получится что в кеш-линию будет грузится куча лишней информации. В этом случае нужно разбить жирный объект на 2 части и хранить его в двух массивах - в одну часть положить поля по которым мы хотим искать и тогда в процессе поиска мы будем юзать 100% данных загруженной кеш-линии а в другую часть объекта положить поля которые нам понадобятся только после уже нахождения нужного объекта (кстати на этом принципе основана работа bitmap-индексов)

Дим
Сейчас в $mol используется css для стилизации. Это имеет следующие недостатки:
1. Нет проверки на этапе компиляции. Можно где угодно написать любой селектор и он может быть применится, а может быть и нет. При рефакторинге эти селекторы нужно править вручную, за чем сложно уследить.
2. Селекторы выглядят довольно монструозно ([mol_app_demo_detail_empty_message], например). Писать их вручную напряжно.

Есть реализация style-in-ts, которая программно генерирует стили, заданные для компонента в TS:
https://github.com/eigenmethod/mol/tree/style-in-ts/style
Там довольно круто всё тайпчекается - по полной используется информация об иерархии и именах компонент. Соответственно, работает и рефакторинг. Местами писанины меньше, местами больше. Правда стили задаются в JSON виде:

$mol_style_define( $mol_page , {
box_shadow: 0 0 0 1px ${ $mol_theme.line }
Body {
overflow: 'scroll',
':hover': {
backgroundColor: $mol_theme.hover,
},
Strut {
display: 'none',
},
},
} )

Оба этих механизма не очень дружат, так как генерируемые стили не могут попасть в правильные места бандла, чтобы переопределения работали корректно.

Поэтому есть 3 пути:
1. Остаться на CSS.
2. Отказаться от CSS.
3. Попытаться как-то их подружить. Например, компилировать генератор стилей отдельно, генерировать при сборке стили и подключать их в правильные места css-бандла. Но это усложнит и замедлит сборку.
