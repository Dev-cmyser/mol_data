В новой реализации свойств я сделал хранение кеша в объектах. Это позволяет задействовать скрытые классы, тем самым ускорив работу и сэкономив память. Однако, это оказалось ошибкой. Теперь я вспомнил, почему в прошлой реализации я хранил кеш в отдельных викмапах. Всё дело в том, что в одном объекте может быть одновременно несколько одноимённых методов и они не должны шарить кеши. Например:

class Parent {
@mem foo() { return 1 }
}

class Child extends Parent {
@mem foo() { return super.foo() + 1 }
@mem bar() { return super.foo() + 1 }
}

То есть кеш должен быть уникальным для комбинации конкретного объекта и конкретной реализации метода. Ну и плюс ключи.

Дим 23.12.2021 22:05:51
Я тут думаю сделать криптосеть. Отличительная особенность моей задумки: это не блокчейн, а конвергенция.

Блокчейн сейчас везде используется в качестве единых монотонных часов. Синхронизировав часы два пира точно знают, какой токен кому принадлежит. Но это приводит к проблеме конкуренции за кворум/монетку, хранения кучи данных, траты кучи ресурсов на майнинг и тп.

Вместо этого можно сделать по настоящему распределённую систему, где каждый хранит лишь часть текущего стейта, а не всю историю с начала времён. Когда надо передать токен, пиры синхронизируют состояния и проверяют возможно это или нет.

Однако, тут надо решить проблему со своевременным распределением изменений по сети, чтобы не было такого, что какой-то умник передаёт токен одному пиру, потом откатывает своё состояние и тут же передаёт другому. То есть оба получателя должны вовремя синхронизоваться, чтобы второй вовремя понял, что ничего не получит.

Но кажется эта проблема может быть решена без блокчейна, за счёт умной выборочной синхронизации через независимые промежуточные узлы по типу DHT. Тут надо поисследовать вопрос.

Например, это может выглядеть так:

- получаем от отправителя изменения с транзакцией токена
- берём часть идентификатора токена в качестве префикса
- находим всех пиров с идентификаторами начинающимися на префикс
- синхронизируем с ними состояние токена
- по завершении синхронизации получаем ответ на вопрос "прошла транзакция или нет"

Фактически это двухфазный коммит получается. Но каждый пир хранит у себя лишь часть токенов: те, что ему интересны, и те, что соответствуют его идентификатору.

Сам же идентификатор пира - это просто публичный ключ, которым можно провалидировать все вносимые в токен изменения.
