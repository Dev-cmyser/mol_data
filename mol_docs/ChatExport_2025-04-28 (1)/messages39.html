Народ, у меня есть идея альтернативной модели модулей (вместо импортов-экспортов) для джаваскрипта. Мне как-то порядком надоело вместе с копированием куска какого-то кода еще отдельно копировать импорты для этого кода. И я подумал что если для кода внутри файла не нужно было бы декларировать импорты отдельно. Тут на ум сразу приходит старый подход когда все файлы подключались тегом <script> в index.html - это действительно избавляет от необходимости явно импортить переменные но с другой стороны неизбежно будут колизии имен и жс выдаст ошибку декларации переменных. Как это можно решить? Давайте вспомним как решается колиззии имен для обычных переменных внутри файла - есть скоупы и понятие вложенности скоупов - и объявление переменной может перекрывать переменную с таким именем в родительском скоупе. Важное преимущество этих скоупов или областей видимости состоит в том что копируя кусок кода из другого файла нам не нужно заботится о коллизиях имен - достаточно заврапить этот код в новый блок и внутри него имена переменных будут перекрывать имена из окружающего кода и тогда этот локальный кусок кода выполнится без ошибок связанных с колиззиями имен.
И идея нового механизма модулей для жс заключается в том чтобы раширить этот механизм вложенности скоупов на файлы и папки. А именно - переменные внутри файла будут доступны из любых других файлов проекта до тех пор пока они не перекрываются другими переменными с таким именем. Если же перекрываются то тогда приоритет берут те переменные которые находятся внутри одной папки рядом с файлом или внутри ближайшей родительской папки (и при необходимости можно добраться до нужных переменных через указание имени файла а потом через точку уже имя самой переменной и эти имена тоже будут подчинятся механизму перекрытия имен) . Как итог - копируя код нам не нужно заодно копировать импорты и сами файлы будут чище без этого полотна импортов и конфликтов тоже не будет так как будет использоваться механизм видимости который аналогичен скоупам внутри файла
